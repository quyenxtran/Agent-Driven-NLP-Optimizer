# Agent Search Policy Notes

Date: 2026-03-20

## Scope

This note summarizes the recent search-policy and execution changes for the SMB agent workflow. It is intended as a stable reference for:

- NC screening behavior
- within-NC optimization behavior
- outer-loop NC rotation logic
- CPU parallelism and timeout policy
- GPT/OpenAI Slurm defaults

## Current Search Design

The workflow now has two explicit levels:

1. Inner loop: screen and optimize within one NC.
2. Outer loop: decide whether to keep investing in the current NC or switch to another NC.

## NC Library

The default NC library is now `all`, not the old 3-layout shortlist.

This means the search is no longer restricted to only:

- `[1,2,3,2]`
- `[2,2,2,2]`
- `[1,3,2,2]`

Instead it uses all valid positive 4-zone partitions generated by the existing `parse_nc_library("all")` path.

## Inner Loop

### Reference screening per NC

Each NC is screened first with reference-style probes.

Default first round:

- `reference`
- `reference_minus`
- `reference_plus`
- `reference_tstep`

Optional extra round:

- `reference_balance`

The intent is:

- keep early evidence conservative and comparable
- avoid contaminating NC screening with aggressive optimization seeds
- let the agent add one extra reference probe when the first round is still too uniformly bad or too uninformative

### Screening run count

The rule of thumb is now:

- minimum per NC: `4`
- maximum per NC: `5`

The 5th reference probe is only used when the first 4 do not produce enough differentiated evidence.

Current environment knobs:

- `SMB_SCREENING_RUNS_MIN_PER_NC`
- `SMB_SCREENING_RUNS_MAX_PER_NC`

### First optimization after screening

Once screening is complete for a given NC:

- the best screening result for that NC is chosen as the flow anchor
- the first optimization run for that NC starts from that anchor

This is deliberate. The first optimization is no longer a blind jump from a generic seed.

## Outer Loop

The outer loop now makes an explicit NC investment decision instead of relying only on flat task ordering.

Current controller behavior:

1. If the current NC still needs reference screening, stay on the current NC.
2. If screening is complete but no optimization has been attempted on the current NC, stay on the current NC and launch the first anchored optimization.
3. If the current NC has feasible or near-feasible optimization evidence, stay on the current NC.
4. If optimization on the current NC is still materially improving relative to the best screening anchor, stay on the current NC.
5. Otherwise, rotate to the next NC that still needs screening.

This gives the workflow a clean separation:

- inner loop: learn the current NC properly
- outer loop: decide when to stop investing in that NC

## Near-Feasible Continuation

If no feasible run exists yet, but the agent finds a run that is near-feasible, the policy now prefers continuation on that same NC before topology-hopping.

Near-feasible is judged using:

- low `normalized_total_violation`
- purity close to threshold
- recovery close to threshold

Current knobs:

- `SMB_NEAR_FEASIBLE_VIOLATION_THRESHOLD`
- `SMB_NEAR_FEASIBLE_PURITY_SLACK`
- `SMB_NEAR_FEASIBLE_RECOVERY_SLACK`

Recommended defaults:

- `SMB_NEAR_FEASIBLE_VIOLATION_THRESHOLD=1e-5`
- `SMB_NEAR_FEASIBLE_PURITY_SLACK=0.005`
- `SMB_NEAR_FEASIBLE_RECOVERY_SLACK=0.005`

## Diagnostic Trigger Refinement

Systematic infeasibility is no longer triggered simply because several runs are `feasible=False`.

Near-boundary low-violation misses are now separated from truly hard-bad runs.

This prevents repeated forcing into diagnostic mode when the search is actually sitting near a feasible basin.

## CPU Parallelism and Time Budgets

### Screening phase

During NC screening, one iteration can execute a same-NC screening bundle in parallel.

Default behavior:

- `4` screening runs for the same NC
- `4` worker processes
- `1` CPU thread per worker
- `180s` maximum solve time per run

This means the workflow can evaluate all 4 reference-style probes for one NC in one iteration.

### Optimization and continuation

Once screening is complete and the policy moves to an optimization or continuation run:

- `1` active solve
- `4` CPU threads
- `300s` maximum solve time

This matches the intended pattern:

- broad parallel screening first
- focused threaded optimization second

Relevant runtime knobs:

- `SMB_IPOPT_WORKERS`
- `SMB_IPOPT_THREADS_PER_WORKER`
- `SMB_SCREENING_IPOPT_THREADS_PER_WORKER`
- `SMB_SCREENING_IPOPT_MAX_SOLVE_SECONDS`
- `SMB_NEAR_FEASIBLE_IPOPT_THREADS_PER_WORKER`
- `SMB_NEAR_FEASIBLE_IPOPT_MAX_SOLVE_SECONDS`
- `SMB_FINALIZATION_IPOPT_THREADS_PER_WORKER`
- `SMB_FINALIZATION_IPOPT_MAX_SOLVE_SECONDS`

## Why the Search Now Behaves Differently

Compared with the older workflow, the search should now behave more like:

1. Pick one NC.
2. Run 4 conservative reference-style probes on that NC in parallel.
3. If those 4 are still too uniform or too uninformative, run 1 extra reference probe.
4. Choose the best screening result as the anchor.
5. Run the first optimization on that same NC using 4 CPU threads.
6. Let the outer loop decide whether to keep pushing that NC or switch to another NC.

This is a more coherent workflow than:

- global reference gating
- flat candidate ordering
- repeated topology hopping before the current NC is actually understood

## Remaining Caveat

The current 4 reference-style probes are broader and safer than before, but they are still fixed probe patterns, not a full adaptive coverage design.

They are good enough for conservative first-round NC screening, but not yet a full NC-aware design-of-experiments generator.

If needed later, the next improvement would be:

- make the first-round reference probes explicitly maximize directional coverage per NC
- choose the optional 5th probe based on the least-covered direction or failure mode
